Main Goal

Build one web-based SOC management platform where alerts from different security tools are collected, normalized, assigned, triaged, escalated, tracked, and reported.

Tools/log sources:

Wazuh
Trend Micro Deep Security
Firewall
EDR/XDR
Email security
Proxy/DNS
Windows/Linux logs
Custom API sources

The system will help:

SOC L1 review and classify alerts.
SOC L2 perform deeper investigation and remediation.
SOC Engineers improve alert quality and detection rules.
SOC Manager monitor SLA, analyst performance, true positive rate, false positive rate, and missed threats.

3. Alert Prioritization Logic

The system should automatically sort alerts using the SOC method you described.

Priority logic:

1. Show only New/Unassigned alerts.
2. Sort by Severity:
   Critical > High > Medium > Low
3. Sort by Oldest Alert Time first.
4. Boost alerts involving critical assets.
5. Boost alerts with known malicious IOC.
6. Boost alerts with multiple correlated events.

Example scoring:

Critical = 100 points
High = 75 points
Medium = 50 points
Low = 25 points

Critical asset = +30
Known malicious IOC = +40
Multiple related alerts = +20
Old alert over SLA = +25

Final priority score:

Alert Priority Score = Severity Score + Asset Score + IOC Score + Correlation Score + SLA Score

Example alerts table:

CREATE TABLE alerts (
    id UUID PRIMARY KEY,
    alert_code VARCHAR(50) UNIQUE,
    source VARCHAR(100),
    name VARCHAR(255),
    description TEXT,
    severity VARCHAR(20),
    status VARCHAR(50),
    verdict VARCHAR(50),
    priority_score INT,
    alert_time TIMESTAMP,
    event_time TIMESTAMP,
    assignee_id UUID,
    asset_id UUID,
    raw_event JSONB,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

Example alert_comments table:

CREATE TABLE alert_comments (
    id UUID PRIMARY KEY,
    alert_id UUID,
    user_id UUID,
    comment TEXT,
    created_at TIMESTAMP DEFAULT NOW()
);

Example tickets table:

CREATE TABLE tickets (
    id UUID PRIMARY KEY,
    ticket_code VARCHAR(50) UNIQUE,
    alert_id UUID,
    title VARCHAR(255),
    severity VARCHAR(20),
    status VARCHAR(50),
    assigned_team VARCHAR(100),
    owner_id UUID,

    resolution_notes TEXT,
    created_at TIMESTAMP DEFAULT NOW(),

15. Alert Quality Score

Each alert should have a quality score.

Example:

Has affected host: +20
Has user: +15
Has source IP: +15
Has destination IP: +15
Has command line/process: +20
Has MITRE mapping: +10
Has triage guide: +10

Result:

Alert Quality Score: 85/100

This helps SOC engineers improve bad alerts.
